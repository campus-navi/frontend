# Git & 협업 전략 (Campus Navi FE 기준)

## 목적
이 문서는 팀 내 Git 사용 방식과 협업 규칙을 통일하여
작업 흐름을 명확하게 하고 충돌과 혼선을 줄이기 위한 가이드입니다.

---

# 빠른 규칙 요약 (필수)

- 모든 작업은 반드시 이슈 생성 후 시작한다
- 브랜치는 반드시 이슈 번호를 포함하여 생성한다
- 하나의 브랜치는 하나의 이슈만 담당한다
- 커밋에는 반드시 (#이슈번호)를 포함한다
- PR 본문에는 반드시 Closes #이슈번호를 작성한다
- 기능 작업과 디자인 작업은 분리한다

이 규칙을 지키지 않으면 PR merge를 진행하지 않는다

---

# 전체 개발 흐름

이슈 생성 → 브랜치 생성 → 개발 → 커밋 → PR → Merge → 이슈 자동 종료

---

# 1. 브랜치 전략

## 기본 원칙

- main 브랜치는 항상 배포 가능한 상태를 유지한다
- 모든 작업은 브랜치에서 진행 후 PR로 병합한다

## 브랜치 네이밍 규칙

type/short-description-issueNumber

## 규칙

- 반드시 이슈 번호를 포함한다
- 하나의 브랜치는 하나의 이슈만 담당한다

## 예시

- feat/signup-ui-12
- fix/login-error-15
- task/apply-design-21

---

# 2. 이슈 전략

## 기본 원칙

- 모든 작업은 반드시 이슈 생성 후 시작한다
- 이슈는 단순 메모가 아니라 작업 명세서 역할을 한다

## 이슈 종류

- feature : 기능 개발
- bug : 버그 수정
- task : 리팩토링, 설정, 디자인 반영 등

---

# 3. 커밋 메시지 규칙

## 기본 형식

type: 작업 내용 (#이슈번호)

## 예시

- feat: 회원가입 페이지 UI 구현 (#12)
- fix: 로그인 에러 처리 수정 (#15)
- task: 회원가입 UI 디자인 반영 (#21)

## 타입 정의

- feat : 기능 개발
- fix : 버그 수정
- task : 일반 작업
- chore : 환경 설정 (필요 시 사용)

---

# 4. PR 전략

## 기본 원칙

- 모든 작업은 PR을 통해 병합한다
- PR은 하나의 기능 단위로 작성한다

## 필수 규칙

PR 본문에 반드시 아래 내용을 포함한다

Closes #이슈번호

PR merge 시 해당 이슈는 자동으로 종료된다

## 필수 체크

- Closes #이슈번호 작성 여부 확인
- 기능 단위 PR인지 확인
- 불필요한 코드 및 로그 제거 확인

## PR 구성 요소

- 작업 내용
- 변경 사항
- 테스트 방법
- 리뷰 포인트

---

# 5. 이슈 연결 전략

## 규칙

- 커밋: (#이슈번호) → 참조만 한다
- PR 본문: Closes #이슈번호 → 이슈를 종료한다

## 예시

커밋:
feat: 회원가입 폼 구현 (#12)

PR:
Closes #12

---

# 6. 작업 단위 기준

## 원칙

- 한 이슈는 하나의 기능 단위로 정의한다
- 한 브랜치는 하나의 이슈만 담당한다

## 좋은 예

- 회원가입 UI 구현
- 이메일 인증 기능 연동
- 로그인 API 연결

## 나쁜 예

- 회원가입 만들기
- 인증 기능
- 프론트 작업

---

# 7. 기능과 디자인 작업 분리

## 기준

- 기능 변경이 포함되면 feature 브랜치를 사용한다
- UI 또는 디자인 변경만 있을 경우 task 브랜치를 사용한다

## 예시

기능 개발:
feat/signup-form-12

디자인 반영:
task/apply-signup-design-21

---

# 8. FE 작업 기준

프론트엔드는 다음 요소를 함께 고려하여 작업한다

- 화면(UI)
- 상태 관리
- API 연동
- 에러 처리
- 사용자 흐름

단순 UI 작업이 아니라 사용자 흐름 단위로 작업을 나눈다

---

# 9. 참고 자료 작성 규칙

## 기본 원칙

- 모든 이슈에는 참고 자료를 반드시 포함한다

## 작성 방식

참고 자료

Figma
- 반드시 프레임 링크를 사용한다 (node-id 포함)

이미지
- 핵심 화면만 첨부한다

## 규칙

- Figma 링크는 필수로 포함한다
- 이미지는 보조 자료로 활용한다

---

# 10. MVP 개발 원칙

- 핵심 기능 중심으로 개발한다
- UI 완성도보다 동작을 우선한다
- 기능 완료 후 디자인을 개선한다

---

# 한 줄 정리

이슈 → 브랜치 → PR → Merge 흐름을 반드시 지킨다
기능과 디자인 작업을 분리한다
커밋은 기록이며 PR은 결과물이다
